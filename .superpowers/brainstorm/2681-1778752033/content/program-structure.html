SP-4 Programme Architecture

How deep should the content structure go? Pick the model that fits how the coach thinks about sessions.

A

Flat — Program → Weeks → Exercises

Each week has a direct list of exercises with sets/reps. No "sessions" or "days" layer. Coach adds exercises to a week, users work through them at their own pace.

Program: Cat C Strength
└─ Week 1: Foundation
   ├─ Squat — 3×5 @ 70% 1RM
   ├─ Bench — 3×5 @ 70% 1RM
   └─ Deadlift — 1×5 @ 75% 1RM
└─ Week 2: Load
   ├─ Squat — 3×5 @ 75% 1RM
   └─ ...

Pros

  • Simple to build
  • Coach fills it in fast
  • Clean user view

Cons

  • No day structure
  • Can't distinguish "Day 1 push / Day 2 pull"
B

Sessions — Program → Weeks → Sessions → Exercises

Each week has named sessions (e.g. Day 1, Day 2, or "Push Day", "Pull Day"). Exercises live inside sessions. This is how real training programmes are structured.

Program: Cat C Strength
└─ Week 1: Foundation
   ├─ Session 1: Lower Body
   │  ├─ Squat 3×5 @ 70%
   │  └─ Deadlift 1×5 @ 75%
   └─ Session 2: Upper Body
      ├─ Bench 3×5 @ 70%
      └─ OHP 3×8 @ 65%

Pros

  • Mirrors real training splits
  • Users complete one session at a time
  • Better % tracking per session

Cons

  • More complex admin UI
  • More DB tables
C

Sessions now, flexible later — ship B but keep it minimal

Build the sessions layer (Option B's structure) but keep the admin UI dead simple for now — no drag-and-drop, just forms. SP-7 (Admin Panel v1) adds the polished builder. The data model is correct from day one.

Same structure as B — but admin creates
sessions via a plain form, not a builder.

Coach: "Add session" → types name →
"Add exercise" → picks from library →
sets sets/reps/% → saves.

Pros

  • Right data model from day one
  • No schema rework in SP-7
  • Functional for the coach immediately

Cons

  • Admin UX is basic (forms only)
  • Larger SP-4 scope than Option A
